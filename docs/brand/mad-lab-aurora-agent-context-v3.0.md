# MAD LAB AURORA — CURRENT AGENT CONTEXT v3.0

**Status:** Canônico operacional para agentes
**Base:** `MANUAL_DA_AGENCIA.md` v3.0.0 + `.aurora/domain-policy.json` v1.1.0
**Escopo:** Direção criativa, design, interfaces, motion, WebGL, sistemas visuais e projetos verticais abaixo do Mad Lab Aurora
**Uso:** Atualização de agentes que ainda operam com premissas antigas de Dark Editorial obrigatório

---

## 1. Definição atual do Mad Lab Aurora

O Mad Lab Aurora é o braço criativo da Aurora.

Ele não é um projeto vertical.

Ele é a camada de criação, experimentação, direção visual, repertório, efeitos, interfaces e experiências digitais onde projetos verticais são concebidos, testados, refinados e promovidos.

O Mad Lab Aurora pode produzir:

* landing pages simples;
* sites institucionais;
* sites editoriais;
* interfaces premium;
* dashboards;
* sistemas operacionais;
* sistemas complexos;
* experiências WebGL;
* organismos cinéticos;
* páginas disruptivas.

A sofisticação não é ponto de partida obrigatório.

A sofisticação é consequência de acúmulo, teste, erro, assimilação e refinamento.

---

## 2. Regra estética absoluta

A única regra estética absoluta do Mad Lab Aurora é:

> Nada que saia do Mad Lab Aurora pode parecer site feito por IA.

Essa regra substitui qualquer premissa anterior que tratava Dark Editorial, luxo invisível, volumetria, física impossível ou qualquer outro estilo como obrigação universal.

Um projeto pode ser claro, escuro, editorial, brutalista, minimalista, comercial, técnico, luminoso, experimental ou operacional.

Mas nunca pode parecer genérico, automático, intercambiável ou produzido por template de IA.

---

## 3. Correção da premissa antiga de Dark Editorial

Dark Editorial não é mais regra-mãe.

Dark Editorial é um repertório possível.

Ele pode ser usado quando o projeto pedir:

* tensão;
* sofisticação;
* silêncio visual;
* autoridade;
* editorialidade;
* contraste dramático;
* atmosfera premium.

Mas ele não deve ser aplicado automaticamente.

O agente deve abandonar a premissa:

> “Todo projeto premium deve ser Dark Editorial.”

Substituir por:

> “Todo projeto deve ter direção visual específica e não parecer IA.”

---

## 4. Física visual como linha de chegada

A quebra das leis da física continua sendo ambição criativa do Mad Lab Aurora.

Mas não é obrigação inicial.

A regra atual é:

1. primeiro clareza;
2. depois composição;
3. depois hierarquia;
4. depois identidade;
5. depois ritmo;
6. depois movimento;
7. depois presença;
8. depois ruptura física.

Uma landing page simples pode ser aprovada sem WebGL, sem Canvas, sem Three.js e sem efeitos avançados.

Mas não pode ser genérica.

A física visual é linha de chegada, não barreira de entrada.

---

## 5. Escala de complexidade

Todo projeto deve ser classificado antes de execução.

### Nível 1 — Página simples

Uso:

* landing page;
* campanha;
* página institucional curta;
* prova de conceito.

Critério:

* clareza;
* intenção visual;
* especificidade;
* zero aparência de IA.

Não exige física avançada.

---

### Nível 2 — Site estruturado

Uso:

* múltiplas páginas;
* navegação;
* CMS;
* cases;
* blog;
* produto narrativo.

Critério:

* consistência;
* design system mínimo;
* ausência de drift visual.

---

### Nível 3 — Interface premium

Uso:

* marca premium;
* experiência comercial;
* storytelling visual;
* motion refinado.

Critério:

* direção de arte;
* ritmo;
* transição;
* tipografia;
* materiais;
* composição intencional.

---

### Nível 4 — Sistema complexo

Uso:

* dashboards;
* CRMs;
* backoffices;
* sistemas internos;
* plataformas operacionais.

Critério:

* usabilidade;
* clareza;
* estados reais;
* performance;
* acessibilidade;
* escalabilidade.

Neste nível, estética nunca pode sacrificar operação.

---

### Nível 5 — Experiência disruptiva

Uso:

* WebGL;
* Three.js;
* Canvas;
* WebGPU;
* organismos visuais;
* scroll cinematográfico;
* interfaces espaciais.

Critério:

* física percebida;
* presença;
* fallback;
* reduced motion;
* performance governada;
* controle de intensidade.

## 7. Relação entre Mad Lab, Elysian, Ozzmosis e verticais

O Mad Lab Aurora governa o craft visual.

Elysian é a marca/identidade mãe quando aplicável.

Ozzmosis é a fábrica/infraestrutura/camada produtiva.

Projetos verticais, como Lex, CRM, ElysianCorp ou outros, consomem diretrizes do Mad Lab, mas podem ter identidade própria.

O agente não deve importar regras de negócio, CRM, funil, preços, clientes ou operação comercial para dentro do Mad Lab.

O Mad Lab produz linguagem visual, sistemas de interface, efeitos, repertório, padrões e pacotes de promoção.

## 8. Gates obrigatórios

Todo output relevante deve passar por gates.

### Design Gate

Valida:

* intenção visual;
* composição;
* hierarquia;
* tipografia;
* cor;
* anti-genericidade;
* ausência de aparência IA.

### Effects Gate

Valida:

* função cognitiva do efeito;
* fallback;
* reduced motion;
* performance;
* controle de intensidade;
* ausência de motion decorativo.

### Web Gate

Valida:

* responsividade;
* acessibilidade;
* performance;
* semântica;
* estados reais;
* integridade de implementação.

### Copy Gate

Valida:

* especificidade;
* ausência de clichês;
* tom adequado;
* CTA concreto;
* narrativa coerente.

---

## 9. Anti-genericidade

O agente deve rejeitar imediatamente qualquer output com sinais de IA genérica.

Sinais bloqueantes:

* hero centralizado com headline, subtítulo e botão genérico;
* três cards iguais com ícones genéricos;
* gradiente roxo/azul SaaS sem justificativa;
* glassmorphism decorativo;
* Lucide icons como identidade;
* Inter/Geist/System UI sem decisão tipográfica;
* CTA genérico como “Saiba mais”, “Começar” ou “Clique aqui”;
* fade-in em todos os elementos sem hierarquia;
* layout que poderia pertencer a qualquer startup;
* visual correto, mas sem gesto autoral.

---

## 10. Liberdade criativa

O agente tem liberdade para propor soluções simples ou complexas.

Liberdade não significa ausência de governança.

Toda solução precisa ter:

* intenção;
* critério de aceite;
* fallback quando houver efeito;
* justificativa visual;
* risco declarado;
* forma de rejeição;
* aprendizado registrável.

---

## 11. Aprendizado acumulativo

Todo projeto produzido pelo Mad Lab deve gerar aprendizado.

O agente deve registrar:

* o que funcionou;
* o que falhou;
* o que pareceu IA;
* o que pareceu genérico;
* quais anti-padrões apareceram;
* quais padrões candidatos surgiram;
* quais erros devem virar vacina;
* o que deve ser preservado para projetos futuros.

Erros não devem ser descartados.

Erros devem virar proteção contra repetição.

---

## 12. Vault local

O conhecimento do Mad Lab entra primeiro no Vault local.

Promoção automática para Vault global é proibida.

Fluxo permitido:

```txt
candidate_ki
→ bibliotecario_sanitization
→ qa_review_gate
→ manual_approval
→ global_vault_promotion
```

`auto_promotion` deve permanecer `false`.

---

## 13. Proibições técnicas

O agente não pode:

* criar gateway MCP local;
* replicar runtime do Alvaro;
* duplicar Qdrant global;
* executar autocura produtiva autônoma;
* burlar gates visuais;
* promover conhecimento automaticamente para Vault global;
* alterar capabilities globais;
* alterar projetos verticais fora do escopo;
* misturar contexto de negócio com direção criativa do Mad Lab.

---

## 14. Motion e efeitos

Todo efeito deve declarar:

1. fenômeno visual/físico;
2. função cognitiva;
3. uso permitido;
4. uso proibido;
5. fallback;
6. budget;
7. critério de rejeição.

Motion decorativo é proibido.

Se um movimento pode ser removido sem alterar feedback, orientação, compreensão ou sensação de presença, ele é decorativo.

---

## 15. Production First

Todo experimento nasce com:

* fallback;
* budget de performance;
* forma de desligar;
* comportamento mobile;
* reduced motion;
* critério de rejection.

Budgets padrão:

```txt
Desktop alvo: 60fps
Mobile alvo: 30fps com fallback
```

Feature flags mínimas:

```ts
enabled: boolean
quality: 'low' | 'med' | 'high'
reducedMotion: boolean
```

## 18. Mantra operacional

Não criamos templates.

Não imitamos IA.

Não confundimos simplicidade com preguiça.

Não confundimos complexidade com qualidade.

Não quebramos a física antes de dominar a composição.

Cada página simples deve ensinar algo.

Cada erro deve virar proteção.

Cada padrão deve nascer testado.

O destino é criar experiências que pareçam vivas.

O primeiro dever é nunca parecer automático.
