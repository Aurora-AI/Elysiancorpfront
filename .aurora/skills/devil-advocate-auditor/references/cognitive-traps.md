# Cognitive Traps — Taxonomia para o Auditor Adversarial

## 1. IKEA Effect
**Definição:** Pessoas atribuem valor desproporcional a coisas que elas próprias construíram, independente da qualidade objetiva.

**Sintomas observáveis no founder:**
- Defende detalhes de implementação que o cliente nunca veria
- Compara o produto com versões anteriores (progresso interno) em vez de com alternativas do mercado
- Interpreta feedback negativo como "o cliente não entendeu", não como "o produto tem problema"
- Investe mais tempo refinando features já prontas em vez de validar se features novas são necessárias

**Antídoto no auditor:** Sempre avaliar o produto pela perspectiva do cliente que nunca viu o processo de construção — apenas o resultado.

---

## 2. Escalada Irracional de Compromisso (Sunk Cost Fallacy)
**Definição:** Continuar investindo em algo que não está funcionando porque já se investiu muito — confundindo investimento passado com indicador de valor futuro.

**Sintomas observáveis:**
- "Já gastei 3 meses nisso — não posso abandonar agora"
- Adiciona features para justificar o trabalho anterior em vez de para resolver problemas reais
- Interpreta persistência como virtude independente da direção
- Dificuldade em pivotar mesmo com sinais claros de que a direção está errada

**Caso clássico:** Quibi — 1.7 bilhões investidos criaram resistência psicológica para pivotar mesmo com todos os dados apontando falha de produto-mercado.

**Antídoto no auditor:** Perguntar "Se você estivesse começando do zero hoje, com o que sabe agora, construiria exatamente isso da mesma forma?" Se a resposta for não, o path atual é sunk cost.

---

## 3. Viés de Confirmação
**Definição:** Tendência de buscar, interpretar e lembrar informações que confirmam crenças pré-existentes.

**Sintomas observáveis:**
- Seleciona os primeiros usuários que sabidamente gostarão do produto
- Interpreta feedback ambíguo como positivo
- Lembra de exemplos de sucesso do mercado que "confirmam" a tese, ignora os fracassos
- Faz perguntas de validação que contêm a resposta: "Você acha que seria útil ter isso, né?"

**Antídoto no auditor:** Para cada dado positivo apresentado, perguntar: "Qual dado negativo não foi mencionado que deveria ter sido?"

---

## 4. Efeito Dunning-Kruger (Invertido — para founders competentes)
**Definição para especialistas:** Quanto mais competente, mais consciente dos próprios blind spots — mas paradoxalmente mais capaz de racionalizar erros sofisticados com argumentos inteligentes.

**A armadilha específica para founders de IA com domínio técnico:**
- Constroem argumentos tecnicamente corretos para conclusões estrategicamente erradas
- Usam complexidade técnica para obscurecer simplicidade do problema de mercado
- Acreditam que profundidade técnica compensa validação de mercado

**Diferença do Dunning-Kruger clássico:** Não é incompetência camuflada. É competência usada para defender o indefensável com eloquência.

**Antídoto no auditor:** Separar a análise em dois planos explícitos: (1) está correto tecnicamente? (2) está correto estrategicamente? Um não implica o outro.

---

## 5. IKEA Effect + Efeito de Dotação (Combinado)
**Definição:** Quando construção própria (IKEA) se combina com posse (Dotação), o valor percebido multiplica — tornando qualquer avaliação objetiva impossível internamente.

**Quando acontece:** Produto está em produção há meses, fundador fez múltiplas iterações, já tem algum investimento de parceiros ou usuários iniciais.

**Sintoma mais perigoso:** O fundador consegue construir narrativas racionais sofisticadas para qualquer dado — não por desonestidade, mas porque o cérebro literalmente processa o produto como extensão de identidade.

**Antídoto no auditor:** Auditoria com critérios definidos *antes* de ver o produto — não depois. O auditor deve formular os critérios de aprovação/reprovação antes de receber o argumento.

---

## 6. Pensamento Probabilístico Ausente (Annie Duke)
**Definição:** Tratar decisões como certas ou erradas em vez de apostas com odds. Buscar certeza onde só existem probabilidades.

**Sintomas observáveis:**
- "Isso vai funcionar" em vez de "Estimo X% de chance de funcionar, dado Y"
- Usa histórias de sucesso (YouTube, Cursor) como provas em vez de como dados em uma distribuição
- Não define critérios de falha antes de iniciar — só reconhece falha quando é óbvio

**Antídoto no auditor:** Forçar estimativa explícita de probabilidade para as suposições principais. Se o founder não consegue dar uma estimativa percentual com justificativa, a suposição é fé, não análise.

---

## Matriz de Diagnóstico Rápido

| Comportamento observado | Viés mais provável |
|---|---|
| Defende implementação mesmo com dados contrários | IKEA Effect |
| Continua direção sem revisar mesmo com sinais de alerta | Escalada irracional |
| Só menciona evidências positivas | Viés de confirmação |
| Argumento sofisticado para conclusão questionável | Dunning-Kruger invertido |
| Não consegue estimar probabilidade de sucesso | Pensamento probabilístico ausente |
| Mistura competência técnica com certeza estratégica | Dunning-Kruger + IKEA combinados |
